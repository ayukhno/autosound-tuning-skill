# Стратегія пресетів — яку бібліотеку пресетів будувати

Один DSP-конфіг = **одна OUTPUT-база (correctness, curve-agnostic)** + **кілька VIRTUAL voicing-пресетів** під різні цілі (механізм base+voicing → `diagnostic-techniques.md §6`, `SKILL.md §Session lifecycle`). Перемикання пресета = заміна **войсингу/налаштувань верхнього шару**, не бази. Тут — *які саме пресети* має сенс заводити і чим вони відрізняються.

> Конкретний набір пресетів — рішення проєкту (з'ясовується на intake: призначення + DSP-ліміт кількості пресетів, `project-intake.md §2/§4`). Нижче — типова бібліотека-орієнтир, не догма.

## Що варіюється МІЖ пресетами (а що ні)
- **База (НЕ варіюється):** кросовери, TA, полярність, фаза стиків, пер-канальна лінеаризація, L/R shape-match. Це «правильний інструмент» — один на всі пресети.
- **Варіюється у пресеті:** войсинг (крива/тілт/шелфи на virtual) · центр/тил on-off · рівень/розширення саба · DSP-FX · **джерело/вхід** · спец-прийоми сцени (crossfeed — нижче).

## Типова бібліотека пресетів
1. **SQ / reference** — точна цільова крива, нейтральний войсинг (Accurate/обрана reference), часто фронт-онлі. Для критичного слухання та суддівства.
2. **FULL / «для себе» / enjoyment** — огортання: центр + тил увімкнені, тепліший войсинг (Laid-back-типу), гарячіший саб. Для щоденного задоволення.
3. **SQL (SQ + Loud)** — як SQ, але з запасом по рівню/headroom і трохи прибраним субсоніком (щоб не бубоніло на гучності). Гучно, але якісно.
4. **Surround / DSP-FX** *(якщо DSP має, напр. Virtual X / sound-field)* — окремий пресет, бо FX міняє фазу/замір (заміри з суфіксом `FX`). **Capability-driven:** з'являється від можливостей заліза — напр. процесор Goldhorn із surround-DSP відкриває surround-пресет. Тому **набір доступних пресетів залежить від обладнання проекту** (з'ясовується на intake §4; досвід по залізу → `knowledge/dsp/`).
5. **Source / input-пресет** — під **інше джерело**: напр. daily-пресет, де **навігація йде з ГУ, а музика — через BT/USB з телефону**. Тут інша маршрутизація входів (а часом і мікс підказок навігації). ⚠️ Пресет тут **прив'язаний до входу** — задокументуй пара «пресет → активний вхід»; перемикання пресета може **тихо скинути вхід** (Pre-session §4, `competition.md`) — найприкріша втрата звуку/балів.

## Competition-пресети — під ПРАВИЛА, не «взагалі»
Різні формати судять по-різному → competition-пресети під **EMMA / AYA / CARMusic різні**, і прийоми можуть бути **взаємовиключні**. Тримати їх окремо, не «один competition».

- **Crossfeed (підмішування L↔R)** — домішати частку протилежного каналу у фронт (трохи L у R і навпаки) → **стабілізує/стискає сцену до центру**, прибирає крайню латералізацію; ціна — менша ширина й сепарація.
  - **EMMA — застосовний** (по практиці користувача): домішування L→R та R→L для **стабілізації сцени**.
  - **AYA — НІКОЛИ** (інша філософія суддівства — цінується природна ширина/сепарація).
  - ⚠️ Це **by-practice варіант**, прив'язаний до конкретного формату й сезону (правила формату міняються — `competition.md`); тримати як свідомий вибір пресета, не як загальне правило. Не плутати з **rear-fill диференційним матриксом L−R** (це на ТИЛ, для огортання — `voicing-by-ear.md §Тил`); crossfeed — на ФРОНТ, для звуження образу.

## Дисципліна
- **Тримати всі пресети одночасно** для A/B на місці свіжими вухами (вибрати під день/категорію/джерело).
- **Лог у changelog:** кожен пресет = назва + ціль + (вхід, якщо специфічний) + спец-прийоми (crossfeed тощо). Іменування `voicing:<intent>` → `naming-and-structure.md §5`.
- **Backup кожного** named-стану у `dsp-config/` (+ README-мапа).
- ⚠️ **Ліміт кількості пресетів** у DSP скінченний (intake §4) — пріоритезувати, які реально потрібні цьому власнику.
